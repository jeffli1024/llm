Philosophy

⚡

Project-First

Every phase starts with a real deliverable. Theory is introduced only when your project forces you to need it — not as prerequisites you read before touching code.

🤖

AI-Augmented

Cursor, Copilot, and Claude are first-class tools. Use them to go faster — but read every generated line. The goal is understanding, not completion.

📊

Measure Everything

Log to Weights & Biases from day one. Intuition without metrics is guessing. This discipline is what makes you credible as a training lead.

✍️

Teach to Learn

After each phase, write a brief explainer. A blog post, an internal doc, a Slack message. Explaining forces precision that reading alone never will.

Phase Breakdown

P.01
Phase 01 · Foundation
Tokens, Embeddings & the Attention Mechanism
3–4 weeks · 2–4 hrs/day
+
🎯 Project Goal Build a character-level language model from scratch — no libraries beyond PyTorch. Tokenize text, implement an embedding layer, write a basic attention head by hand, and train the model to predict next characters on a small corpus (e.g. Shakespeare). You will generate text before you read a single paper.

Theory — introduced as you need it

  • Tokenization: BPE vs character-level — why it matters when batching
  • Embeddings: what the weight matrix actually learns
  • Scaled dot-product attention: derive Q, K, V from first principles
  • Softmax temperature: see it control randomness directly in your output
  • Backprop through time — only when your gradients vanish in practice

Stack & Tools

  • PyTorch (raw — no HuggingFace yet)
  • Weights & Biases for experiment tracking
  • Cursor IDE with Claude/GPT-4o for pair-coding
  • Jupyter notebooks for visualising attention matrices
  • GitHub — commit daily, even small experiments

Hands-on Tasks (in order)

  • 01Set up Python 3.11, PyTorch, Jupyter, W&B, GitHub repo — get the environment solid first
  • 02Download tiny-shakespeare; write a BPE tokenizer from scratch in under 100 lines
  • 03Implement single-head self-attention with no library — validate every tensor shape
  • 04Stack 2–4 transformer blocks; add positional encoding
  • 05Train on CPU first, then free Colab GPU; log loss curves to W&B
  • 06Generate text — compare greedy vs temperature vs nucleus sampling
  • 07Use Cursor to accelerate boilerplate; manually review every AI suggestion before running it
KEY MATH Attention(Q,K,V) = softmax( QKᵀ / √d_k ) · V
Derive it. Implement it. Visualise the attention weights as a heatmap on your Shakespeare text.
MILESTONE Your character-level GPT generates recognisable Shakespearean prose. You can explain every tensor shape in the forward pass.
P.02
Phase 02 · Architecture
GPT-2 Architecture Deep Dive
4–5 weeks · 2–4 hrs/day
+
🎯 Project Goal Go from toy model to real GPT-2 architecture (124M parameters). Implement multi-head attention, layer norm, MLP blocks, and train on a real dataset (OpenWebText subset). Use Karpathy's nanoGPT as a reference — but instrument and extend every part of it yourself.

Theory — when you hit it in nanoGPT

  • Multi-head attention: why multiple heads and what each learns
  • LayerNorm vs BatchNorm: stability during deep training
  • Residual connections: see the effect by removing them experimentally
  • GPT-2 weight init: std=0.02, scaled by layer depth
  • Cosine annealing with warmup: derive the schedule, see it in W&B

Stack & Tools

  • nanoGPT (Karpathy) as base — read every line first
  • Lambda Labs / Vast.ai for GPU rental (~$1/hr A100)
  • W&B Sweeps for hyperparameter search
  • Cursor for refactoring experiment variants
  • Flash Attention (implement manually, then compare)

Hands-on Tasks (in order)

  • 01Clone nanoGPT — read and annotate every single line before running anything
  • 02Instrument with W&B: log grad norms, weight histograms, per-head attention entropy
  • 03Rent an A100 and run a training job on a 10B token OpenWebText subset
  • 04Implement Flash Attention manually; benchmark it against naive attention
  • 05Run ablations: remove residuals, change init, change LR — document what breaks and why
  • 06Use Claude to explain any confusing GPT-2 paper section; cross-check explanation with code
  • 07Write an internal blog post explaining the architecture to a junior dev from memory
KEY MATH Parameter count formula: ~12 × n_layers × d_model² ≈ 124M for GPT-2 small.
Verify this yourself by summing named_parameters() and matching the formula.
MILESTONE You can explain every tensor shape in a GPT-2 forward pass from memory, and you have W&B logs from a real training run you own.
P.03
Phase 03 · Data Engineering
Building Domain-Specific Training Data Pipelines
3–4 weeks · 2–4 hrs/day
+
🎯 Project Goal Data quality is the single biggest lever in LLM training. Build an end-to-end pipeline: raw web crawl → deduplication → quality filtering → tokenisation → streaming dataloader. This phase leverages your existing SE skills the most — lean into it.

Theory — introduced when the pipeline breaks

  • MinHash LSH deduplication: why duplicates hurt more than you think
  • Quality filters: perplexity scoring, n-gram heuristics (C4 paper)
  • Chinchilla data-compute scaling: the key insight about tokens per param
  • Tokenizer training: SentencePiece, tiktoken, vocab size tradeoffs
  • Streaming datasets: why you can't load 1TB into RAM

Stack & Tools

  • Apache Spark or Polars for large-scale processing
  • datasketch library for MinHash LSH
  • SentencePiece, tiktoken
  • HuggingFace datasets (streaming mode)
  • AI tools for pipeline scaffolding; profile for bottlenecks

Hands-on Tasks (in order)

  • 01Download a 10GB CommonCrawl WARC dump; build an extraction and cleaning pipeline
  • 02Implement MinHash deduplication; measure the exact deduplication rate on your corpus
  • 03Train a SentencePiece tokenizer on your domain corpus; compare different vocab sizes
  • 04Build a streaming DataLoader with prefetching; profile I/O vs GPU utilisation
  • 05Reproduce C4 quality filters; compare perplexity of a small model trained before vs after
  • 06Use AI to generate Spark/Polars boilerplate; profile and find the bottleneck
KEY MATH Chinchilla optimal: N_tokens ≈ 20 × N_params
Understand why — and more importantly, when to deviate from it for your specific use case.
MILESTONE A fully reproducible pipeline that turns a raw web crawl into a clean, deduplicated, tokenised dataset ready for training.
P.04
Phase 04 · Infrastructure
Distributed Training at Scale
4–5 weeks · 2–4 hrs/day
+
🎯 Project Goal This is where you earn the title of training lead. Configure and run distributed training with DDP, Tensor Parallelism, and Pipeline Parallelism on a real multi-GPU setup. Profile GPU utilisation, identify bottlenecks, and achieve greater than 50% Model FLOP Utilisation (MFU).

Theory — when your GPUs sit idle

  • Data Parallel (DDP): gradient AllReduce — when it becomes the bottleneck
  • Tensor Parallelism (Megatron): split weight matrices across GPUs
  • Pipeline Parallelism: micro-batching to hide bubble overhead
  • Mixed precision: FP16 vs BF16 vs FP8, loss scaling explained
  • ZeRO optimizer stages: shard optimizer state, gradients, parameters
  • Gradient checkpointing: trading compute for memory

Stack & Tools

  • PyTorch DDP + FSDP
  • DeepSpeed ZeRO (stages 1, 2, 3)
  • Megatron-LM for Tensor Parallelism
  • NVIDIA Nsight Systems for profiling
  • W&B for distributed run monitoring
  • Lambda Labs / CoreWeave 4×A100 node

Hands-on Tasks (in order)

  • 01Set up a 4×A100 node on Lambda; configure passwordless SSH and shared storage
  • 02Train a 1.3B model baseline with DDP; measure GPU utilisation using nsys + PyTorch profiler
  • 03Add gradient checkpointing; measure the memory vs speed tradeoff empirically
  • 04Integrate DeepSpeed ZeRO-2; compare memory footprint before and after
  • 05Implement Tensor Parallelism using Megatron-LM; write unit tests to verify numerical correctness
  • 06Profile and optimise until you hit >50% MFU — document every change and its effect
  • 07Write a runbook: how to detect, diagnose, and resume after a node failure mid-training
KEY MATH MFU = achieved_FLOPS / peak_hardware_FLOPS
Model FLOPS per forward+backward pass ≈ 6 × N_params × N_tokens
Calculate this for your 1.3B run before running it. Then measure the gap.
MILESTONE A 1.3B model training run you fully own: you can start it, monitor it, debug it when it crashes, and resume it from checkpoint without losing progress.
P.05
Phase 05 · Alignment
Fine-Tuning, RLHF & Direct Preference Optimisation
4–5 weeks · 2–4 hrs/day
+
🎯 Project Goal Take an open pre-trained model (Llama-3 7B) and make it genuinely useful for a specific domain — instruction following in code review, medical Q&A, or a domain you choose. Run full SFT, then apply DPO as a modern, simpler alternative to PPO-based RLHF.

Theory — introduced when your model ignores instructions

  • SFT: data format, loss masking on prompt tokens (critical detail)
  • RLHF: reward model training, PPO — conceptual understanding
  • DPO: why it's simpler than PPO; the preference pair data format
  • LoRA / QLoRA: low-rank adaptation math — why rank 16 works well
  • Catastrophic forgetting: regularisation and data replay strategies

Stack & Tools

  • HuggingFace TRL (SFTTrainer, DPOTrainer)
  • PEFT library (LoRA, QLoRA)
  • vLLM for fast inference during evaluation
  • GPT-4o as LLM-judge for automated win-rate eval
  • AI tools for data curation assistance

Hands-on Tasks (in order)

  • 01Curate 10K instruction pairs for your domain; use AI to assist but manually review 500 samples
  • 02Run full SFT on Llama-3 7B with HuggingFace TRL; verify loss masking is correct on prompt tokens
  • 03Implement LoRA fine-tune; compare adapter size vs quality vs full fine-tune
  • 04Collect 2K preference pairs (chosen vs rejected responses) for your task
  • 05Run DPO training; measure win rate vs SFT baseline using LLM-as-judge
  • 06Run QLoRA on a single A100; understand the memory vs quality tradeoff quantitatively
KEY MATH DPO loss: −log σ( β·log(π_θ(y_w|x)/π_ref(y_w|x)) − β·log(π_θ(y_l|x)/π_ref(y_l|x)) )
Understand what β controls, what π_ref is, and why no reward model is needed.
MILESTONE A fine-tuned 7B model that measurably outperforms the base Llama-3 on your domain task, with win-rate numbers you can defend.
P.06
Phase 06 · Production
Evaluation, Inference Optimisation & Deployment
3–4 weeks · 2–4 hrs/day
+
🎯 Project Goal A model no one can use is worthless. Build a rigorous evaluation harness, quantise the model for efficient inference, and deploy it as a scalable API with full observability. This is what separates a researcher from a training lead who ships.

Theory — when you hit latency or cost walls

  • Eval: BLEU/ROUGE vs LLM-judge vs task-specific — when to use each
  • Quantisation: INT8 vs INT4 vs GPTQ vs AWQ tradeoffs
  • Speculative decoding: draft model + verification, acceptance rate math
  • KV cache: why it grows linearly; PagedAttention (vLLM) solution
  • Throughput vs latency: the batching tradeoff for serving

Stack & Tools

  • lm-evaluation-harness (EleutherAI)
  • vLLM for production inference
  • AutoGPTQ, llm-awq for quantisation
  • Grafana + Prometheus for monitoring
  • FastAPI as serving wrapper

Hands-on Tasks (in order)

  • 01Build an eval harness using lm-evaluation-harness on 5+ benchmarks; establish baselines
  • 02Implement GPTQ and AWQ quantisation; measure perplexity degradation at each bit width
  • 03Deploy with vLLM; load test with 100 concurrent requests; measure p50/p99 TTFT and throughput
  • 04Add OpenTelemetry tracing; build a Grafana dashboard: tokens/sec, TTFT, error rate, GPU util
  • 05Write a model card: training data, eval results, known failure modes, intended use, limitations
  • 06Set up automated regression evals: any change that degrades benchmarks >1% triggers an alert
KEY MATH Throughput = batch_size × tokens_per_step × steps_per_second
Know your A100 peak FLOPS (312 TF/s BF16) and calculate theoretical max throughput before measuring.
MILESTONE A production API with latency SLOs, a Grafana dashboard, automated regression evals, and a model card you'd be proud to share publicly.

Essential Papers — Read When You Hit the Concept

Phase
Paper
Why It Matters
P.01
Attention Is All You Need — Vaswani et al. (2017)
The transformer architecture you are building. Read alongside your code, not before it.
P.02
GPT-2 Technical Report — Radford et al. (2019)
Architecture details and hyperparameters for the training run you will replicate.
P.03
Chinchilla — Hoffmann et al. (2022)
Reframes everything: it's about tokens-per-parameter, not just model size.
P.04
Megatron-LM — Shoeybi et al. (2019)
Tensor parallelism design — read while implementing Phase 4's distributed setup.
P.04
ZeRO — Rajbhandari et al. (2020)
How to shard optimizer state. The key insight for training models larger than a single GPU's memory.
P.05
InstructGPT — Ouyang et al. (2022)
The original RLHF pipeline. Understand it before moving to DPO so you know what DPO simplifies.
P.05
DPO — Rafailov et al. (2023)
The simpler RLHF alternative you will implement. Short paper, high-value derivation.
P.06
Flash Attention 2 — Dao (2023)
Why naive attention is slow. The IO-aware algorithm that makes training tractable at scale.

Training Lead Competency Map

Technical Decisions

  • Architecture selection and parameter sizing
  • Tokenizer and training data mix design
  • Parallelism strategy (DP / TP / PP tradeoffs)
  • Hyperparameter schedule design
  • Evaluation benchmark selection
  • Quantisation and serving strategy

Operational Ownership

  • Debug a diverging training loss systematically
  • Resume from checkpoint after node failure
  • Identify GPU bottlenecks (compute vs I/O vs comms)
  • Manage cloud GPU cost against training schedule
  • Maintain reproducible experiment logs
  • Set up automated regression eval pipelines

Team & Communication

  • Write clear, actionable experiment reports
  • Review ML engineer PRs with real depth
  • Brief stakeholders on training status honestly
  • Estimate compute budget before a new run
  • Write model cards with honest limitations
  • Onboard new engineers to the training stack

Daily & Weekly Rhythm

MON – WED

2–3 hrs hands-on coding on the phase project. Use AI tools freely. Commit working code to GitHub daily, even if it's small.

THURSDAY

1–1.5 hrs reading the week's relevant paper section or math concept — only the parts your code actually touched.

FRIDAY

1 hr reviewing W&B logs, writing a short experiment summary, and updating your learning notes in a running doc.

WEEKEND

Optional. Watch Karpathy's YouTube series, explore tangents, run experiments that came up during the week.

You Are Ready to Lead When...

You can estimate the GPU-hours and cost of a new training run from first principles — before touching a single line of code.

When a training loss diverges at step 40K, you have a systematic debugging playbook and can pinpoint the root cause.

You can write the attention mechanism from scratch on a whiteboard, explaining every design decision as you go.

You can look at a W&B dashboard for a training run you didn't launch and identify the three most important things to investigate.

A junior engineer asks why their fine-tuned model keeps repeating phrases. You diagnose the root cause in under 10 minutes.

You can brief a non-technical executive on "why we need to retrain" with confidence, numbers, and a clear recommendation.

Build things. Measure everything. Ship models.
